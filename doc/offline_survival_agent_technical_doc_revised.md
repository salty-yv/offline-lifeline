# 离线自救助手技术文档

版本：v0.3  
更新方向：聚焦离线模型、LiteRT-LM 生命周期管理、对话式 Agent、轻量本地工具  
目标平台：Android 中高端设备  
核心模型：Gemma-4-E2B-it-litert-lm  
推理框架：LiteRT-LM

---

## 1. 项目概述

离线自救助手是一款面向 Android 手机的本地对话式生存 Agent。项目目标是在断网、低电量、户外受困、自然灾害、受伤、无法搜索信息等场景下，让用户依靠手机本地模型获得保守、可执行的自救建议。

本项目采用默认内置模型的离线发行方式。用户安装应用后，无需首次联网下载模型，即可在无网络环境下直接使用。应用内置 Gemma-4-E2B-it-litert-lm 模型，通过 LiteRT-LM 在 Android 端完成本地推理，并结合对话式 Agent Runtime、极简 Safety Kernel、本地应急工具和少量离线参考资料，为用户提供多轮对话、图片辅助理解、行动计划生成和工具推荐能力。

软件的ui可以切换英文或中文

项目核心定位：

> 一个安装即用、完全离线运行的 Android 对话式生存 Agent。

用户可以通过文字和图片描述当前处境。Agent 会主动追问关键信息，判断风险，生成分阶段行动建议，并在必要时引导用户使用本地工具，例如 SOS、屏幕求救、电量保护、本地应急信息卡和离线指南。

---

## 2. 产品边界

本项目不是在线搜索工具，也不是医疗诊断工具，更不是完整救援调度系统。项目优先解决的是：用户在无法依赖网络时，仍能通过本地模型获得安全、保守、可执行的行动建议。

项目明确不做：

- 云端问答；
- 在线搜索；
- 在线医生问诊；
- 自动报警；
- 完整救援调度系统；
- 社交社区；
- 复杂路线规划；
- 复杂户外导航；
- 面向低端设备的性能承诺；
- 让模型判断未知野外食物一定安全；
- 让模型替代专业救援或医疗服务。

项目优先级：

> 先完成离线模型推理、对话式 Agent、Safety Kernel 和轻量本地工具，再逐步扩展图片、多场景测试和性能优化。

## 3. 核心设计原则

### 3.1 安装完成即具备离线能力

应急类应用最重要的问题之一是可靠性。用户可能在出发前忘记下载资源，也可能在真正需要时没有网络。因此，本项目默认采用内置模型的离线发行方式。

应用安装后，模型文件、基础应急参考资料和核心工具应随包准备完毕。首次启动时，应用只完成模型文件检查、完整性校验和运行时初始化，而不是要求用户联网下载大模型。

首次启动联网下载模型可以作为轻量商店版、测试版或备用发行方案，但不作为主路径。

### 3.2 Agent 是对话式的

本项目的核心体验不是“用户问一句，系统答一句”，而是“用户和一个离线生存 Agent 持续协作”。

当用户描述情况不完整时，Agent 应主动追问。例如用户说“我在山里迷路了，脚扭了，天快黑了”，Agent 不应只给一段泛泛建议，而应立即完成三件事：

1. 给出不需要更多信息也能执行的立即行动；
2. 追问影响决策的关键信息，例如电量、能否移动、是否有水和保暖物；
3. 根据用户后续回答持续调整计划。

### 3.3 模型主导，安全约束兜底

Gemma-4-E2B-it-litert-lm 承担主要的理解、推理、追问、组织和生成工作。项目不应退化为厚重模板 App。

但在蛇咬、误食、严重出血、昏迷、胸痛、呼吸困难、失温、中暑、洪水、雷暴、火灾等高风险场景中，系统必须有强制安全边界。Safety Kernel 不替 Agent 回答，而是限制 Agent 不能建议危险动作，并要求在特定风险场景中包含必要提醒。

### 3.4 工具能力优先服务行动

应急场景下，用户需要的不是长篇解释，而是下一步该做什么。Agent 的回答应尽量导向行动，并在合适时推荐本地工具。

MVP 阶段的本地工具应保持轻量，避免引入实现成本高、调试复杂、受设备差异影响大的能力。优先实现独立、稳定、可离线使用的工具，例如：

- SOS 闪光灯；
- 屏幕高亮 SOS；
- 电量保护建议；
- 本地个人应急信息卡；
- 离线指南入口；
- 本地 Debug Log 导出。

## 4. 核心应用场景

### 4.1 户外迷路

用户在深山、林地、荒野中迷路，手机无信号，只剩少量电量。Agent 应通过多轮对话确认电量、身体状态、装备、水、保暖物、天色、同行人员等信息，并给出停止盲目移动、保存体力、避风保温、制作求救信号、节省电量等建议。

### 4.2 受伤处理

用户摔伤、扭伤、割伤、烧伤或出现出血情况。Agent 应先判断是否存在严重风险，追问出血量、疼痛程度、是否能移动、是否有麻木、变形、意识模糊等信息，并给出保守的初步处理建议。

### 4.3 图片求助

用户拍摄伤口、虫咬、植物、蘑菇、河流、天气现象、设备故障或现场标识。Agent 通过本地多模态模型理解图片内容，并结合用户文字描述进行对话式分析。

图片能力应定位为辅助理解和风险分流，而不是确定性诊断。例如，面对蘑菇图片，Agent 不应判断“可以食用”，而应建议未知野外蘑菇不要食用，并给出替代求生策略。

### 4.4 自然灾害

用户遭遇地震、洪水、火灾、暴雨、雷电、泥石流等情况。Agent 应根据灾害类型、用户当前处境、是否被困、是否受伤、是否有安全撤离条件等信息，提供避险建议、撤离注意事项、求救信号方法和灾后自救提醒。

### 4.5 低电量求生

用户手机电量不足。Agent 应优先保护关键使用能力，减少不必要的模型推理，引导用户关闭高耗电功能、降低屏幕亮度、减少连续对话、准备求救信息、使用离线指南和 SOS 工具。

## 5. 总体技术架构

项目采用 Android 原生技术栈，核心架构包括：

- Android App；
- Kotlin；
- Jetpack Compose；
- LiteRT-LM Runtime；
- Gemma-4-E2B-it-litert-lm 内置模型；
- LiteRT-LM 生命周期管理模块；
- 对话式 Agent Runtime；
- CameraX 图片输入；
- Room / SQLite 本地存储；
- SQLite FTS 离线检索；
- Safety Kernel 极简安全内核；
- 本地应急工具模块；
- DataStore 设置存储；
- 本地 Debug Log。

整体分层如下：

1. **UI 层**  
   负责对话界面、图片输入、工具入口、设置页面和状态提示。

2. **Agent 层**  
   负责上下文管理、风险判断、追问规划、行动计划生成、工具推荐和安全约束注入。

3. **推理层**  
   负责 LiteRT-LM Runtime 初始化、模型加载、会话创建、流式输出、推理中断和资源释放。

4. **安全层**  
   负责高风险场景约束、危险建议拦截、必要提醒注入和输出前校验。

5. **数据层**  
   负责对话历史、应急信息卡、设置项、模型状态、离线指南和本地日志存储。

6. **工具层**  
   负责 SOS 闪光灯、屏幕高亮 SOS、电量保护建议、离线指南入口、应急信息卡和 Debug Log 导出。

## 6. 模块设计

### 6.1 对话式 Agent Runtime

Agent Runtime 是本项目相较传统离线问答工具的关键差异。它负责理解用户输入、维护多轮上下文、追问缺失信息、制定行动计划，并在用户持续反馈后调整建议。

Agent Runtime 包含以下子模块：

#### 6.1.1 Context Manager

维护多轮对话上下文，包括：

- 用户当前问题；
- 已知场景信息；
- 用户身体状态；
- 电量状态；
- 装备和物资；
- 已经给出的建议；
- 尚未确认的关键信息；
- 当前行动计划。

Context Manager 需要控制上下文长度，避免无限堆叠历史消息导致推理变慢或超出模型上下文窗口。

#### 6.1.2 Risk Classifier

判断当前对话涉及的风险域，例如：

- 迷路；
- 受伤；
- 低电量；
- 夜间；
- 失温；
- 中暑；
- 脱水；
- 蛇虫咬伤；
- 误食；
- 洪水；
- 火灾；
- 雷暴；
- 地震后建筑物风险。

Risk Classifier 的输出会影响 Safety Kernel 约束、追问内容和工具推荐。

#### 6.1.3 Question Planner

判断是否需要主动追问，并生成少量关键问题。追问原则：

- 一次不要问太多；
- 优先问会影响下一步行动的问题；
- 信息不足时也要先给可执行的立即行动；
- 对高风险场景优先确认生命体征和紧急程度。

#### 6.1.4 Action Planner

生成阶段性行动建议，建议结构包括：

- 当前风险判断；
- 现在立刻做什么；
- 接下来几分钟做什么；
- 不要做什么；
- 需要继续确认的问题；
- 什么时候必须求救。

Action Planner 应尽量输出短句、步骤化内容，避免大段科普。

#### 6.1.5 Tool Router

判断是否应该推荐本地工具。MVP 阶段推荐的工具包括：

- SOS 闪光灯；
- 屏幕高亮 SOS；
- 电量保护建议；
- 本地个人应急信息卡；
- 离线指南入口；
- Debug Log 导出。

Tool Router 只负责判断和推荐，不把工具调用做成复杂自动化流程。这样可以降低实现难度，也能避免用户在紧急场景中被复杂交互打断。

#### 6.1.6 Safety Adapter

将 Safety Kernel 生成的约束注入模型上下文，限制危险建议，并要求模型在必要场景中包含关键提醒。

Safety Adapter 的职责是把安全规则转成模型能理解的上下文约束，而不是替代模型生成完整回答。

---

### 6.2 LiteRT-LM 生命周期管理

LiteRT-LM 是本项目端侧推理的核心运行时。为了保证离线 Agent 在 Android 设备上的稳定性，架构中必须明确 LiteRT-LM 的初始化、会话管理、推理中断、异常恢复和资源释放流程。

LiteRT-LM 生命周期管理模块负责以下工作。

#### 6.2.1 启动前检查

应用启动 Agent 前，应先完成基础检查：

- 检查模型文件是否存在；
- 校验模型文件完整性；
- 检查模型清单版本；
- 检查可用存储空间；
- 检查模型资源是否处于可加载状态；
- 检查上一次异常退出状态。

如果模型文件缺失或校验失败，应阻止进入模型对话，并展示降级提示，让用户仍可打开离线指南和本地工具。

#### 6.2.2 Runtime 初始化

Runtime 初始化流程：

1. 读取模型资源清单；
2. 定位内置模型文件路径；
3. 初始化 LiteRT-LM Runtime；
4. 加载 Gemma-4-E2B-it-litert-lm 模型；
5. 创建本地推理会话；
6. 注册流式输出回调；
7. 初始化推理状态管理器；
8. 将推理会话交给 Agent Runtime 使用；
9. 标记离线 Agent 已就绪。

初始化过程应在 UI 中展示明确状态，例如“正在检查模型”“正在加载模型”“Agent 已就绪”。如果初始化耗时较长，应避免界面假死。

#### 6.2.3 会话创建与复用

为了减少重复加载开销，应用应在一次连续对话中复用同一个推理会话。

会话复用策略：

- 同一轮连续对话复用当前 session；
- 用户清空对话时重置 session；
- 上下文过长时压缩历史或重建 session；
- 模型异常时销毁 session 并尝试重新创建；
- App 进入后台时根据资源压力决定保留或释放 session。

会话复用应与 Context Manager 配合，避免把全部历史原样塞回模型导致性能下降。

#### 6.2.4 流式输出管理

LiteRT-LM 输出应优先采用流式展示，降低用户等待感。

流式输出管理需要处理：

- token 增量回调；
- UI 渲染节流；
- 用户主动停止生成；
- 生成完成状态；
- 生成失败状态；
- Safety Kernel 输出前校验；
- 部分输出被拦截后的重试或改写。

高风险场景下，如果 Safety Kernel 需要完整回答后再校验，可以先缓存模型输出，校验通过后再展示，或采用“安全前缀 + 后续流式”的折中方案。

#### 6.2.5 推理中断

用户退出当前回答、切换页面、点击停止生成、设备过热或电量过低时，应支持中断正在进行的推理。

推理中断要求：

- 能取消当前生成任务；
- 不破坏 Runtime 状态；
- 不让 UI 停留在“生成中”；
- 不把未完成回答误写入最终历史；
- 允许用户重新发起对话；
- 必要时提示“本次生成已停止”。

推理中断是 MVP 中必须验证的稳定性能力，因为用户在紧急场景中很可能反复打断或切换操作。

#### 6.2.6 资源释放

资源释放必须写入架构设计，不能只作为实现细节处理。

需要释放的资源包括：

- LiteRT-LM 推理会话；
- 模型句柄；
- 推理缓存；
- 图片输入临时数据；
- 流式输出回调；
- 后台推理任务；
- 与模型状态绑定的 UI 引用。

资源释放触发条件：

- 用户关闭 Agent 页面；
- 用户主动清空会话；
- App 退出；
- 系统内存压力过高；
- 模型运行异常且无法恢复；
- 设备进入极低电量状态；
- 长时间无对话活动。

释放完成后，应更新模型状态，避免后续误认为模型仍处于可用状态。再次进入 Agent 时，应重新执行必要的初始化流程。

#### 6.2.7 异常恢复

LiteRT-LM 相关异常包括：

- 模型文件缺失；
- 模型校验失败；
- Runtime 初始化失败；
- 会话创建失败；
- 推理过程中断；
- 内存不足；
- 图片输入处理失败；
- App 切后台后恢复失败。

异常恢复策略：

- 初始化失败：展示降级提示，保留离线指南和工具箱可用；
- 推理失败：终止当前生成，允许用户重试；
- 会话损坏：释放旧 session，尝试重新创建；
- 图片失败：降级为文字对话；
- 内存不足：清理上下文、释放图片缓存、提示用户重启 Agent；
- 多次失败：进入安全降级模式，只开放非模型能力。

### 6.3 内置模型离线发行

本项目默认采用内置模型的离线发行包。

发行包应包含：

- Android 应用主体；
- Gemma-4-E2B-it-litert-lm 模型文件；
- 极简本地参考资料库；
- 模型和资源清单；
- 完整性校验信息；
- 必要初始化配置。

首次启动流程：

1. 检查模型文件是否存在；
2. 校验模型文件完整性；
3. 检查可用存储空间；
4. 初始化 LiteRT-LM Runtime；
5. 创建本地推理会话；
6. 初始化 Agent Runtime；
7. 标记离线 Agent 已就绪。

模型版本、资源清单和校验信息应尽量集中管理，避免后续升级时出现资源不一致问题。

### 6.4 图片辅助理解

图片模块通过 CameraX 拍照或从相册导入图片。图片输入进入模型前，应进行压缩、尺寸限制和隐私处理，例如去除 EXIF 信息。

图片回答原则：

- 先描述图片中可见内容；
- 不做确定性医疗诊断；
- 不判断未知植物、蘑菇或野果一定可食用；
- 不鼓励用户进行高风险操作；
- 对严重风险给出保守建议；
- 必要时追问更多现场信息。

图片输入应接入 LiteRT-LM 生命周期管理，确保图片临时数据在推理完成、中断或失败后及时释放。

---

### 6.5 本地应急工具箱

本地应急工具箱不依赖网络，也不应完全依赖模型。它既可以作为独立入口，也可以由 Agent 在对话中推荐用户打开。

MVP 阶段建议包含：

- SOS 闪光灯；
- 屏幕高亮 SOS；
- 电量保护建议；
- 本地个人应急信息卡；
- 离线指南入口；
- 本地 Debug Log 导出入口。

工具箱的价值在于即使模型加载较慢、用户电量不足或用户不想继续对话，也能直接执行关键操作。

---

### 6.6 极简本地参考资料

本项目不采用大量知识卡片作为主回答路径，而是保留一个极简本地参考资料库。它主要用于：

- 给 Agent 提供少量事实锚点；
- 支持离线指南浏览；
- 覆盖高风险场景的基础提醒；
- 在模型不可用时提供降级内容；
- 为 Safety Kernel 提供必要标准建议。

参考资料优先覆盖：

- 迷路；
- 出血；
- 扭伤与骨折；
- 烧伤；
- 失温；
- 中暑；
- 脱水；
- 蛇虫咬伤；
- 误食；
- 雷电；
- 洪水；
- 火灾；
- 地震；
- 求救信号；

## 7. 对话流程

典型对话流程：

1. 用户输入文字或图片；
2. Agent 读取当前对话上下文；
3. Agent 判断当前场景和风险域；
4. Safety Kernel 根据风险域生成约束；
5. Agent 判断是否缺少关键信息；
6. 如果信息不足，Agent 同时给出立即行动和少量追问；
7. 如果信息足够，Agent 生成阶段性行动计划；
8. Tool Router 判断是否推荐本地工具；
9. LiteRT-LM 生成回答；
10. 输出前进行安全校验；
11. 用户继续反馈，Agent 更新计划。

这种流程能够让系统保持对话感，而不是退化为静态指南或单轮问答。

## 8. Safety Kernel 设计

### 8.1 设计目标

Safety Kernel 是项目的极简安全内核。它不是厚重知识库，也不是替代模型回答的模板系统。它的目标是让 Agent 保持模型主导和对话式体验，同时避免在高风险场景中输出明显危险的建议。

Safety Kernel 只做三类事情：

1. 禁止危险动作；
2. 要求必要提醒；
3. 输出前校验。

### 8.2 高风险场景

Safety Kernel 至少覆盖以下风险域：

- 严重出血；
- 昏迷；
- 呼吸困难；
- 胸痛；
- 溺水；
- 蛇咬；
- 失温；
- 中暑；
- 严重过敏；
- 误食植物、蘑菇、药物；
- 烧伤；
- 骨折；
- 儿童、老人、孕妇、慢病患者相关场景；
- 洪水；
- 雷暴；
- 火灾；
- 地震后建筑物风险；
- 夜间迷路；
- 低电量求生。

### 8.3 禁止输出方向

模型输出中如果出现以下倾向，应被拦截、改写或要求重新生成：

- 鼓励食用未知植物、蘑菇、野果；
- 建议吸吮蛇毒；
- 建议自行切开伤口；
- 建议放血或用危险方式处理伤口；
- 建议喝酒保暖；
- 建议严重受伤时继续冒险移动；
- 对图片做确定性医疗诊断；
- 对野外食物做确定性安全判断；
- 鼓励涉水穿越洪水；
- 鼓励雷暴中躲在孤立大树下；
- 在胸痛、呼吸困难、昏迷等场景中弱化求救必要性。

### 8.4 安全和 Agent 的关系

本项目不采用“高风险场景直接模板回答”的主路径。更合适的方式是：

- Agent 仍然负责理解、追问和生成；
- Safety Kernel 负责注入边界；
- 输出前由安全校验器检查危险建议；
- 如果不合格，要求模型重新生成或由系统改写关键句。

这样既保留 Agent 项目的大模型属性，又避免模型在高风险情境中自由发挥到危险方向。

## 9. 本地存储与隐私

本项目默认本地运行，用户输入、图片和推理结果不上传云端。

本地存储包括：

- 对话历史；
- 用户应急信息卡；
- 设置项；
- 模型状态；
- 本地 Debug Log；
- 离线参考资料。

隐私设计原则：

- 图片处理前去除 EXIF 隐私信息；
- Debug Log 默认不上传；
- 用户可清除对话历史；
- 应急信息卡应支持本地锁定或隐藏敏感字段；
- 模型推理、图片处理和对话记录默认只在设备本地完成。

## 11. 设备与性能假设

本项目不以低端设备为主要目标。推荐设备条件包括：

- Android 中高端或高端设备；
- 64 位 ARM 架构；
- 较新 SoC；
- 足够的可用存储空间；
- 建议 8GB RAM 或更高；
- 支持较长时间本地推理；
- 可接受大安装包和较高存储占用。

需要在真实设备上验证：

- App 安装成功率；
- 首次启动初始化耗时；
- 模型完整性校验耗时；
- Runtime 初始化耗时；
- 模型加载耗时；
- 首 token 延迟；
- 平均生成速度；
- 图片输入延迟；
- 连续对话后的内存占用；
- 资源释放是否彻底；
- 推理中断是否稳定；
- 发热和降频；
- 低电量模式下表现；
- 飞行模式下全流程可用性。

## 12. MVP 范围

### 12.1 MVP v0.1：内置模型 + 对话式文字 Agent

目标：完成安装即用的离线文字 Agent 闭环。

功能包括：

- 内置 Gemma-4-E2B-it-litert-lm 模型；
- LiteRT-LM 本地加载；
- LiteRT-LM Runtime 初始化；
- LiteRT-LM 推理中断；
- LiteRT-LM 资源释放；
- 飞行模式下首次打开可用；
- 文字对话输入输出；
- 多轮上下文管理；
- Agent 主动追问；
- 阶段性行动计划生成；
- 极简 Safety Kernel；
- 本地离线指南入口；
- 本地 Debug Log。

验收标准：

用户安装完成后，不联网、不下载模型，直接开启飞行模式。用户输入“我在山里迷路了，脚扭伤，天快黑了”，Agent 能主动追问电量、能否移动、是否有水和保暖物，并立即给出安全、简洁、可执行的前三步行动建议。

### 12.2 MVP v0.2：Agent 工具推荐

目标：让 Agent 能够在对话中推荐用户使用本地工具，而不是只输出文字建议。

功能包括：

- SOS 闪光灯；
- 屏幕高亮 SOS；
- 电量保护建议；
- 本地个人应急信息卡；
- 离线指南入口；
- Agent 根据上下文推荐工具。

验收标准：

用户输入“我迷路了，手机只有 12% 电”，Agent 能优先建议停止不必要操作、降低屏幕亮度、减少连续对话、准备关键求救信息，并引导用户使用 SOS 或离线指南等本地工具。

### 12.3 MVP v0.3：图片对话 Agent

目标：支持图片 + 文字的多模态离线对话。

功能包括：

- CameraX 拍照；
- 相册图片导入；
- 图片压缩；
- 去除 EXIF 信息；
- 图片 + 文字输入；
- 图片内容描述；
- 图片风险识别；
- 多轮追问；
- 图片场景下的 Safety Kernel 校验；
- 图片输入相关临时资源释放。

验收标准：

用户拍摄伤口时，Agent 输出保守急救建议并追问风险信息；用户拍摄蘑菇时，Agent 不判断其可食用；用户拍摄河流时，Agent 提醒水流、低温、污染和涉水风险。

## 13. 主要风险与应对

### 13.1 Agent 输出错误建议

大模型可能产生幻觉或危险建议，尤其是在医疗、误食、蛇咬、洪水、雷暴等场景中。

应对策略：

- 建立 Safety Kernel；
- 覆盖禁止动作和必要提醒；
- 输出前做安全校验；
- 建立红队测试集；
- 对图片诊断和野外食物判断保持保守。

### 13.2 LiteRT-LM 运行稳定性

端侧模型可能出现初始化慢、内存占用高、生成中断失败、切后台恢复失败或资源释放不彻底等问题。

应对策略：

- 单独设计 LiteRT-LM 生命周期管理模块；
- 明确 Runtime 初始化、session 复用、中断和释放流程；
- 在真实设备上测试内存峰值和长对话稳定性；
- 推理异常时允许重建 session；
- 模型不可用时保留离线指南和工具箱。

### 13.3 应急产品责任边界

项目可能涉及健康、急救和灾害建议，存在合规和责任风险。

应对策略：

- 明确产品是应急信息辅助工具；
- 不宣称医疗诊断能力；
- 不替代专业救援和医生；
- 在高风险场景中建议尽快联系救援；
- 隐私政策说明本地数据处理方式。

### 13.4 安装包体积和设备门槛

内置模型会显著增加安装包体积，并要求用户设备具备足够存储和运行能力。

应对策略：

- 主路径接受大安装包；
- 明确推荐设备条件；
- 保留轻量发行方案作为备用；
- 首次启动进行资源检查；
- 模型不可用时提供降级能力。

## 14. 验收指标

| 指标 | 目标 |
|---|---|
| 安装后离线可用 | 安装完成后开启飞行模式仍可启动 Agent |
| 模型完整性校验 | 首次启动能校验模型文件 |

| 资源释放 | 退出 Agent 后释放 session、缓存和临时资源 |
| 文字对话闭环 | 支持多轮上下文和主动追问 |
| 立即行动建议 | 信息不足时仍能先给安全行动建议 |
| 工具推荐 | Agent 能根据上下文推荐本地工具 |

## 15. 一句话总结

离线自救助手是一款基于 Android、LiteRT-LM 和 Gemma-4-E2B-it-litert-lm 的本地对话式生存 Agent。它默认内置模型，安装后即可离线运行，用户可以通过多轮对话和图片输入描述处境，Agent 会主动追问关键信息、分析风险、制定行动计划，并推荐 SOS、电量保护、应急信息卡和离线指南等本地工具，在极简 Safety Kernel 约束下提供保守、可执行的自救建议。
